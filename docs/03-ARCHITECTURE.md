# Architecture

Монорепозиторий: `apps + packages + data`.

Главное правило проекта: **единственное хранилище данных — JSON**. PostgreSQL, MySQL, MongoDB и другие БД не используются.

Система разделена на независимые контуры доступа, работающие с общей JSON-моделью:

1. публичный сайт, PWA и Telegram Mini App;
2. внутренняя CRM только для сотрудников компании (`owner`, `admin`, `manager`);
3. внешний партнёрский кабинет для одобренных прямых партнёров;
4. CPA API и адаптеры сетей для автоматической работы с вебмастерами CPA-сетей.

Арбитражники и вебмастера не получают доступ к внутренней CRM. Менеджеры компании ведут клиента, меняют статусы, добавляют комментарии и причины отказа. Эти внутренние изменения формируют внешние CPA-события и обезличенную партнёрскую статистику.

Общий контур:

```text
переход CPA/реф → расчёт → заявка → внутренняя CRM → статус менеджера → очередь CPA → адаптер сети → S2S-postback
```

Прямой партнёр видит только обезличенные переходы, расчёты, заявки, договоры, баланс и собственные запросы выплаты. CPA-сеть получает технические события через свой адаптер и рассчитывается с вебмастерами внутри сети самостоятельно.

Для высоконагруженных коллекций используется чанкинг: максимум 500 записей в одном JSON-файле. После заполнения создаётся следующий файл и индекс коллекции.

Основные высоконагруженные коллекции:

```text
data/leads/leads.json
data/clients/clients.json
data/cpa/events.json
data/activity/feed.json
```

Индекс создаётся автоматически рядом с коллекцией:

```text
leads-index.json
events-index.json
```

Запись выполняется через временный файл с заменой, чтобы уменьшить риск повреждения JSON при записи.
